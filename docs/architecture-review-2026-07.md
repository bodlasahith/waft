# Local Council — Waft app assessment

**Question:** Perspectives on (1) architecture decisions, (2) technology choices, (3) likely debugging dead-ends / fragile areas, (4) security tradeoffs, and (5) overall novelty and growth potential of the Waft app.

**Date:** 2026-07-21 · **Mode:** local council (no external providers configured)

> **Local council** — these perspectives all come from Claude playing different
> roles, not from different AI vendors. Treat agreement as a shared starting
> point to pressure-test, not as independent confirmation.

## 🗳️ Devil's Advocate

### Position
Waft is a two-database, four-runtime, five-integration architecture wrapped around ~1,700 lines of actual API code — the complexity budget has been spent on infrastructure and branding while the core differentiators ("clusters become group chats," per-event subgraphs, RLS-backed data safety) are, in the code as written, either thin veneers or actively broken. I'd argue for collapsing the stack before adding a single feature.

### Key points
- **Neo4j is résumé-driven complexity, and it splits your source of truth.** Connections exist *only* in Neo4j (`packages/api/src/services/graph.ts`) while profiles/socials live in Postgres — so every user write, avatar change, and account deletion spans two databases with no transaction, coordinated by ordering comments ("graph node first: if it fails…", `routes/users.ts:179`). `DELETE /users/me` destroys the graph node *first*; if the Postgres delete then fails, the user's edges are gone forever but the account survives — the code comment claims retryability, but the graph data is unrecoverable. At event scale (hundreds of nodes), a Postgres `edges` table with a recursive CTE does everything the Cypher here does, transactionally, in one database.
- **The event-subgraph model is lossy by design.** `createConnection` stores a *single* `eventId` per edge and `SET r.eventId = coalesce($eventId, r.eventId)` lets a rescan at event B silently steal the edge from event A's graph (`graph.ts:48`). "Time-bounded subgraphs" is the pitch; the schema can't actually represent an edge belonging to two events. Meanwhile `isWallExpired` returns `false` when `ends_at` is null (`routes/events.ts:18-21`) — so any event without an end date is a **permanent, unauthenticated public attendee list** (`GET /events/:eventId/graph` and the WS `/events/:eventId/live` have no auth), directly contradicting the comment above the route.
- **The flagship feature is mostly deep-link theater.** Of five "group chat" platforms (`routes/groups.ts`), only Discord is automated — and it creates a channel in *one shared hub guild* with no permission overwrites, meaning every Waft group is visible to every other member of that guild, with a 7-day unlimited-use invite. Telegram returns a "please create the group yourself" link, WhatsApp returns a list of phone numbers "copied to your clipboard," Slack a multi-DM link. That's fine for a demo, but "graph clusters become group chats" is currently a marketing sentence, not a capability.
- **The rate-limit design fails exactly at its target venue.** The comment in `index.ts:83-87` claims "a whole event of attendees each get their own bucket via trustProxy" — false on conference WiFi, where the entire venue NATs to one public IP and shares one 200-req/min bucket. Every check-in also triggers a full-graph Neo4j query plus a broadcast of the *entire* graph snapshot to every WS subscriber (`liveEvents.ts`, `routes/events.ts:142-143`): O(N) payload × O(N) viewers at peak arrival time, on an in-memory subscriber map that assumes a single Railway instance and evaporates on every deploy. The system's worst-case load profile *is* the demo (YC AI Startup School).
- **`infra/schema.sql` contains zero RLS statements, and the anon key ships in the mobile binary.** `apps/mobile/src/supabase.ts` embeds `EXPO_PUBLIC_SUPABASE_ANON_KEY`; without `enable row level security` on `users`, `social_links`, and `feedback`, anyone extracting that key can read every email, phone number (stored as a `phone` social link — the input to WhatsApp groups), and private social handle via PostgREST, bypassing all the careful `visibility` logic in `routes/users.ts`. Maybe RLS was clicked on in the dashboard — but the repo's infra-as-code says otherwise, and the API's visibility gating is worthless if the side door is open. (ARCC was not queried — MCP server unavailable — standard guidance applied.)

### Risks & blind spots
- **Duplicate graph renderers**: Sigma.js/Graphology on web, but a hand-rolled O(N²)-per-tick force layout over `react-native-svg` on mobile (`GraphScreen.tsx`) — two divergent viz codebases to debug, and the mobile one will chug on a JS thread at real event sizes.
- **Config drift is a live landmine**: mobile encodes QR origin `https://waft.app` (`config.ts`) while CORS allows `https://getwaft.app` and email sends from `getwaft.app` — whichever domain is actually canonical, one half of the system disagrees. Mobile auth also runs the deprecated `implicit` OAuth flow because PKCE broke on-device — a workaround comment, not a resolution.
- **`getNetworkGraph` interpolates depth into Cypher** (`[:WAFT*1..${depth}]`) and runs unbounded `shortestPath((origin)-[:WAFT*]-…)` per node — clamped to 4, but a negative `?depth=-1` produces a Cypher error → 500, and on a dense post-conference graph this query's cost explodes; Aura free-tier pausing/throttling will make this look like random flakiness during debugging.
- **Zero tests, anywhere.** Every invariant above is enforced by comment, not by CI. The next refactor of `coalesce($eventId, …)` or the delete ordering breaks silently.
- **Novelty question nobody's asking**: this is Linktree-QR + a party-trick graph wall, in a graveyard of dead predecessors (Bump, CardDrop, Hume badges…). The defensible asset would be the cross-event social graph — the one thing the current schema mangles (single `eventId` per edge) and the one thing users can vaporize for everyone via cascade-free `DETACH DELETE`. Growth potential depends on the graph compounding; right now it can't.

### Confidence
`high` — every claim above is grounded in specific lines read in the repo, not inference; the only hedged item is RLS, which could exist outside version control.

## 🗳️ Simplicity Champion

### Position
Waft's core loop (scan → card → edge → live graph) is a few thousand lines of genuinely simple code wrapped around one large unforced complexity: a second database. Neo4j Aura is the biggest over-engineering in this repo — every query it runs is 1–2 hops that a single Postgres table would handle, while the dual-store split forces hand-rolled distributed-transaction reasoning throughout the API.

### Key points
- **Neo4j earns nothing here; it costs a lot.** I read every Cypher query in `/Users/bsahith/Documents/waft/packages/api/src/services/graph.ts` — MERGE a person, MERGE an undirected edge, list neighbors, list event attendees, "are A and B connected", "do A and B share an event". All of it is one `connections(user_a, user_b, event_id, created_at)` table plus one join away in the Postgres you already run. The tell is in `/Users/bsahith/Documents/waft/packages/shared/src/analytics.ts`: the actual graph analytics (Brandes betweenness, density, top connectors) are computed **client-side in ~100 lines of shared TS** because "Aura Free has no GDS library" — so the graph database contributes zero graph-y value. Even `distance` in `getNetworkGraph` is computed server-side with an unbounded `shortestPath((origin)-[:WAFT*]-(connected))` per node, when the client already receives all nodes+edges and could BFS it for free (it already runs Brandes on the same data). Dropping Neo4j deletes a vendor, ~250 lines of session boilerplate, and an entire failure class.
- **The dual store forces saga logic in a CRUD app.** `/Users/bsahith/Documents/waft/packages/api/src/routes/users.ts` is full of careful ordering comments: "Graph node first: if it fails, no row is written… (Row first would strand a profile with no graph node — connections to it 404 forever)"; account deletion touches three systems (Neo4j → Postgres → Supabase auth) with per-step idempotency reasoning. Name, photo, and avatar are duplicated into both stores ("mirrored onto the Neo4j Person node so every graph response carries it without a Supabase join") — guaranteed drift. `connections.ts` then has to stitch event names from Supabase onto edges from Neo4j at read time. All of this is complexity you wrote *because* of the second database, not despite it. One Postgres transaction replaces every one of these comments.
- **Two divergent graph-viz stacks for the same graph.** Mobile hand-rolls a force layout + SVG renderer (~600 lines, `/Users/bsahith/Documents/waft/apps/mobile/src/screens/GraphScreen.tsx` — honest, deterministic, fine). Web pulls in Sigma.js + Graphology — a WebGL engine built for tens of thousands of nodes — to place ~100 event-wall dots at `Math.random() * 100` positions with no layout at all (`/Users/bsahith/Documents/waft/apps/web/src/components/EventGraph.tsx`, 86 lines). Sigma is résumé-driven dependency weight here; the mobile layout code in `packages/shared` would serve both surfaces.
- **The 5-platform group-creation matrix is mostly instructions text.** In `/Users/bsahith/Documents/waft/packages/api/src/routes/groups.ts` + `services/platforms/`, only Discord is "automated," and even that is just a channel in one shared hub guild (Discord blocks bot-created guilds) with best-effort DMs. Telegram returns a deep link telling the user to create the group themselves; WhatsApp returns "Open WhatsApp, create a new group, and add the listed members." That's five integrations of maintenance surface delivering what is, for 4 of 5, a copy-paste workflow. Ship Discord (or just a "copy handles" button) until an event proves demand.
- **Speculative and cosmetic weight before launch.** `strength` is plumbed through Cypher, API responses, shared types, and edge rendering, yet is permanently `1` ("strength will grow… see roadmap") — a roadmap field taxing every layer today. Meanwhile the wordmark animation is ~740 lines across two near-duplicate components (`CoalesceWordmark.tsx` 365 + `CoalesceLogo.tsx` 378), and recent commits tune flutter constants (`RIP_A=0.6`) — more code than the connections route that is the product. Credit where due: `liveEvents.ts` (in-memory Map pub/sub, full-snapshot broadcasts), the empty-body parser fix, and the well-commented rate-limit/CORS setup are exactly the right size.

### Risks & blind spots
- **"But Neo4j is the moat/scaling story."** The in-memory WebSocket subscriber Map in `liveEvents.ts` already pins the API to a single process — you are architecturally single-instance regardless, so Aura buys no scale headroom that the rest of the stack can cash. If Waft someday needs real multi-hop analytics at scale, migrating an edge table out of Postgres then is a well-trodden path; carrying dual-store consistency bugs into your launch event is not.
- **Simplification I would *not* make:** the three-tier social visibility (`public/event_only/mutual_only`) looks like premature ACL complexity — two extra graph round-trips per card view in `sendPublicCard` — but it's a privacy promise, not gold-plating; keep it (it just gets cheaper in Postgres: one query instead of two Neo4j sessions).
- What my lens under-weights: the unauthenticated `/events/:eventId/live` WebSocket and Cypher `${depth}` string interpolation (clamped, but a wart) are security-lens items; I flag them only because removing Neo4j and moving live-graph auth into one store makes both simpler to fix. Note: ARCC was not queried (MCP server unavailable); standard practices applied for the security-adjacent observations.
- Fragility my lens predicts: the first debugging dead-ends will be cross-store ghosts — a user present in Postgres but not Neo4j (or vice versa) after a partial registration/deletion, surfacing as `createConnection` returning `null` → mysterious 404 "One or both users not found" at the worst moment: mid-event.

### Confidence
`high` — the codebase is small (~6.7k lines) and I read all load-bearing files; every Neo4j query is enumerable and none exceeds what a single relational table expresses, which makes the central claim checkable rather than a matter of taste.

## 🗳️ Security Auditor

ARCC note: The `search_arcc` MCP tool is not registered in this environment, so ARCC was not queried (MCP server unavailable). Per the governance rules I proceeded with standard security practices and grounded everything below in the actual code.

### Position
The core auth model is actually well-considered — identity always comes from the verified Supabase JWT, never the request body — but the app leans on a single hand-rolled authorization layer with no database backstop, and that layer is already inconsistent: the group endpoints leak PII (phone numbers) that the card endpoint deliberately protects. That inconsistency, plus the service-role key bypassing RLS, is the security story here.

### Key points
- **The group routes bypass the visibility model the card routes enforce.** `sendPublicCard` (users.ts:38-56) carefully filters `social_links` by `visibility` (`public` / `event_only` / `mutual_only`). But `/groups/create` and `/groups/suggest-platform` (groups.ts:67-71, 142-154) query `social_links` by `user_id`+`platform` with **no visibility filter at all**. A single QR scan makes you "connected," and `filterConnectedUsers` is the only gate — so one scan lets you pull a person's raw phone number via a WhatsApp/iMessage group even if they set that phone link to `event_only` and never shared an event with you. Two code paths, two different answers about who may read the same PII. `suggest-platform` even returns the actual `handles` array in its `coverage` payload (groups.ts:87,102).
- **Service-role key = no defense in depth.** `lib/supabase.ts` uses `SUPABASE_SERVICE_KEY`, which bypasses Row Level Security, and `infra/schema.sql` defines **zero RLS policies**. Every authorization decision lives in app code as manual `.eq("id", req.userId)` scoping. That works today because most queries remember to scope — but there is no second layer. One forgotten filter on any future route is a full-table read of users/socials/events. For a product whose whole value is aggregated PII and a social graph, single-layer authz is the biggest structural risk.
- **The live event surface is fully unauthenticated and low-entropy-gated.** `/events/:eventId/graph` (HTTP) and `/events/:eventId/live` (WebSocket, index.ts:109) require no auth, and `/events/by-code/:code` (public) resolves an 8-char nanoid code straight to the event id and attendee graph (names). Anyone who sees/guesses a shared event code can enumerate the room's attendee list and open unlimited WS subscriptions — `@fastify/rate-limit` does not meaningfully cover WS upgrades, and there's no origin check on the socket. Attendee-name disclosure + connection-flood DoS at exactly the target use case (a conference).
- **The rate limit is both spoofable and self-defeating at events.** The comment in index.ts:16-20 is candid that `trustProxy: true` trusts a client-spoofable `X-Forwarded-For` — so any authenticated abuser rotates that header for unlimited buckets, defeating the limit. Worse, the comment's premise is backwards for the actual use case: a whole event on shared conference NAT presents **one** public IP, so all attendees collide into a single 200/min bucket and legitimately DoS each other. The anti-spam control is weakest precisely at YC AI Startup School scale.
- **Smaller but real:** (a) The feedback honeypot is dead code — `website: z.string().max(0)` (feedback.ts:11) makes any filled honeypot fail validation with a `400 invalid_request` *before* the silent-drop `if (website)` at line 25 ever runs, so bots get a distinguishing error instead of the intended silent 200. (b) `/connections` (connections.ts:23) writes any caller-supplied `eventId` onto the WAFT edge with no check that the caller attended it, letting a non-attendee inject edges into an event's public graph. (c) The web card renders `social.url` directly into an `<a href>` (c/[code]/page.tsx:46-48); Zod's `.url()` accepts `javascript:` scheme, so a user-controlled `javascript:` link reaches an anchor href — relying on React's version-dependent sanitization rather than an explicit http(s) allow-list.

### Risks & blind spots
- The "novelty/growth" lens will celebrate the frictionless one-scan connection, but frictionless is the vulnerability: **scanning is the sole authorization event** for both graph edges and (via the group leak) PII access, and QR codes at events are trivially shoulder-surfed, photographed off a badge, or reused. There is no consent step, no per-connection scope, and no rotation trigger — `card_code` rotation is described as possible in a comment (users.ts:83) but no endpoint implements it.
- Privacy-by-time-expiry (`WALL_TTL_HOURS = 24`) is not a security control — the graph is fully public for the entire event window, and account deletion (`DELETE /users/me`) removes rows but any attendee who already scanned or opened a group has exfiltrated the socials/phone already. GDPR/CCPA "right to deletion" doesn't reach data already handed to third parties.
- Other lenses will treat the service-role-key + no-RLS choice as a reasonable simplicity win. From the security lens it removes the one backstop that would make a future authz bug non-catastrophic. That tradeoff is invisible until the first mistake.
- No secrets-handling review was possible for actual key storage (Railway env), and ARCC could not be consulted — if there is org guidance on service-key usage or public endpoint exposure, it wasn't checked.

### Confidence
`high` — findings are read directly from the route handlers and schema; the group-visibility bypass and the absent-RLS/service-key model are concrete and verifiable, not speculative.

## 🗳️ Scalability Architect

### Position
Waft's current architecture is a well-built single-box demo that will work beautifully at a 150-person meetup and fall over at exactly its stated target — a YC-AI-Startup-School-sized event — because the live layer is pinned to one process, the broadcast model is O(viewers × graph) full snapshots, and the hot Neo4j query does per-row unbounded `shortestPath`. The good news: the event is a natural shard key, so the fix path is clear.

### Key points
- **The live layer cannot horizontally scale — at all.** `packages/api/src/lib/liveEvents.ts` is a process-local `Map<eventId, Set<WebSocket>>`, and `@fastify/rate-limit` uses its default in-memory store (`packages/api/src/index.ts:92-96`). Add a second Railway instance and broadcasts from a check-in handled on instance A never reach viewers whose socket landed on instance B — the wall silently freezes for half the room. Scaling past one process requires Redis pub/sub (or Supabase Realtime) for fanout and a shared rate-limit store. Until then, capacity = one Node process.
- **Full-snapshot broadcast makes event load quadratic in attendance.** Every check-in and every connection re-runs `getEventGraph` (two Neo4j queries) and pushes the *entire* graph JSON to *every* subscriber (`routes/events.ts:139-143`, `routes/connections.ts:32-33`) — check-in even broadcasts twice. At a 2,000-person event with a door-open burst: N check-ins × N viewers × a payload that grows with N. There is no debouncing, no coalescing, no deltas. A 2-second coalesced snapshot per event, or `{type:"node_added"}` deltas, cuts this by orders of magnitude and is the single highest-leverage change in the repo.
- **`getNetworkGraph` is a combinatorial time bomb** (`services/graph.ts:60-101`). `MATCH path = (origin)-[:WAFT*1..depth]-(connected)` enumerates *paths*, not nodes — in the dense clique-like graphs this product deliberately creates, path counts explode; then it computes an **unbounded** `shortestPath((origin)-[:WAFT*]-(connected))` per returned row. Depth is capped at 4, but depth 4 on a post-conference graph is effectively the whole network with no `LIMIT` on nodes or edges. Aura will start timing these out well before 10x. Use a bounded BFS (`apoc.path` or GDS) that returns node + distance in one pass.
- **Per-IP rate limiting keys on exactly the wrong dimension for this product.** With `trustProxy`, all attendees on the venue's NAT'd WiFi present the same egress IP — the whole conference shares 200 req/min buckets at precisely the moment of peak legitimate traffic. Authenticated routes should key on `req.userId`; only anonymous routes should key on IP.
- **Synchronous work sits in the request path that belongs in async jobs.** `POST /events` blocks on an Anthropic LLM call for icebreakers (`routes/events.ts:38-41`); check-in blocks on a full graph recomputation before responding. Every WS connect also runs a Supabase query + `getEventGraph` — a reconnect storm (venue WiFi blip) is a thundering herd straight into Neo4j with no cache; a 2–5s TTL cache on the event graph would absorb nearly all of it.

### Risks & blind spots
- **The scaling unit is the event, and events are spiky, not gradual.** "10x growth" here isn't 10x DAU over a year — it's 0→3,000 concurrent users in ten minutes when one big conference adopts you, then back to zero. Average-load reasoning (and Railway's single-instance-friendly defaults) will pass every test until the one event that matters. The upside other lenses may miss: `eventId` is a clean partition key — event rooms are independent subgraphs, so a per-event fanout channel and cache shards naturally.
- **Dual-store writes (Supabase + Neo4j) have no outbox/queue.** User creation and account deletion each touch both stores in sequence; deletion is thoughtfully idempotent, but at scale partial failures will accumulate as drift (Person nodes without users, or vice versa) with no reconciliation job.
- **The web client has no WS reconnect** (`apps/web/src/app/event/[eventId]/page.tsx:45-61` — no `onclose` handler, and the cleanup closure is never even wired into the effect), so any blip permanently freezes the wall; when you do add reconnect, add jittered backoff or you've built the thundering herd yourself.
- **Client-side rendering scales worse than the server.** Sigma.js re-ingests a full snapshot per message and `computeGraphStats` reruns on every update on every viewer's phone — at 1,000+ nodes the wall dies in the browser even if the API survives.
- Per project governance: ARCC was not queried (MCP server unavailable); standard practices applied.

### Confidence
`high` — I read the actual hot paths (WS fanout, broadcast sites, Cypher queries, rate-limit config), and the single-instance and O(N×graph) findings are directly visible in the code rather than inferred.

## 🗳️ Maintainability Advocate

### Position
The Waft codebase is unusually well-narrated for a pre-launch app — the comment discipline is genuinely excellent — but every load-bearing invariant lives *only* in prose: there is not a single test, no CI, no lint config, and no type contract between API and clients. The current author can maintain this; the second developer (or the same author in six months) is walking into a minefield of silently-breakable ordering rules.

### Key points
- **Zero tests, zero CI, zero lint — verified, not assumed.** No `*.test.*` files, no vitest/jest config, no `.github/workflows`, no eslint/prettier anywhere in the repo, and no workspace even has a `test` script. Meanwhile the code is full of order-dependent invariants that comments carefully explain but nothing enforces: `DELETE /users/me` in `/Users/bsahith/Documents/waft/packages/api/src/routes/users.ts` must delete graph node → orphan events → Postgres row → auth record *in that order* for retryability; `POST /users` must create the Neo4j node *before* the Postgres upsert and must preserve `card_code` or printed QR codes die. These are exactly the invariants a refactor breaks without failing loudly — the visibility-entitlement logic in `sendPublicCard` (public/mutual_only/event_only) is a five-line block whose regression would be a privacy leak, and its only guardrail is a comment.
- **The dual-store (Postgres + Neo4j) write path has no consistency mechanism — the app's core maintainability tax.** Every user-mutating route does two unrelated writes with no transaction, saga, or reconciliation job. `graph.ts` even documents the failure mode ("connections to it 404 forever"). Future debugging dead-end: a user who exists in Supabase but not Neo4j (or vice versa) after a partial failure, with no tooling to detect or repair the drift. Half the comments in the codebase exist to route around this design; a repair/reconcile script or an outbox would eliminate a whole class of them.
- **`packages/shared` is a contract that nobody honors.** `apps/mobile/src/api.ts` hand-redeclares `Avatar`, `PublicCard`, `WaftEvent`, and both graph payload shapes instead of importing them; worse, the shared types are camelCase (`photoUrl`, `startsAt` in `UserCard`/`EventInfo`) while the API actually returns snake_case Supabase rows (`photo_url`, `starts_at`) — so the "shared types" are partly fiction and actively misleading to a new contributor. There's no schema-derived client (the Zod schemas already exist server-side and could be the single source of truth). API/client drift is currently detected by runtime crashes on a phone.
- **Naming and duplication debt in the brand layer.** The wordmark component is called `Coalesce*` (an old codename?) while the product is Waft — `CoalesceLogo.tsx` (web, 378 lines) and `CoalesceWordmark.tsx` (mobile, 365 lines) are two parallel hand-tuned implementations of the same flutter animation, with the magic constants (`A=0.6, k=6.8, ω=2.6`) tri-plicated across both files and DESIGN.md prose. Next brand tweak gets made in one place and drifts in the other two. Similarly `GraphScreen.tsx` (597 lines) mixes hand-rolled physics, gesture handling, platform-badge styling tables, and data fetching in one file — the single hardest file to modify safely.
- **Real strengths worth preserving deliberately.** The why-comments (trustProxy XFF caveat, the `errorResponseBuilder` 429-vs-500 pitfall in `index.ts`, `filterConnectedUsers`'s anti-harvesting rationale) are the best I've seen in a repo this size, routes/services/lib layering in `packages/api` is clean, and files are mostly small. The right move is cheap: a vitest setup with ~10 tests pinning exactly the invariants the comments already articulate. The specs are literally written; they just aren't executable.

### Risks & blind spots
- **The comments create false confidence.** Reviewers (and other council lenses) will read this code and rate it highly because it *explains itself* — but explanation is not enforcement. This repo's failure mode is a confident edit that contradicts a comment nobody re-read.
- **Two silent scale/refactor traps documented nowhere:** `liveEvents.ts` is in-process `Map`-based pub/sub — the moment Railway runs 2 replicas, live graph updates half-vanish nondeterministically (a classic multi-day debugging dead-end); and `getNetworkGraph` string-interpolates `depth` into Cypher and runs unbounded `shortestPath((origin)-[:WAFT*]-…)` — the route clamps depth to 4, but the function accepts any number, so the safety lives at the wrong layer and one new call site removes it.
- **Expo churn is acknowledged but under-managed.** `AGENTS.md` says "Expo HAS CHANGED, read v57 docs" — good instinct, but that's a note to AI agents, not a lockfile-plus-upgrade-playbook; RN 0.86/React 19.2 pinning via root `overrides` is fragile across workspace boundaries.
- Security/architecture lenses will likely praise the auth and privacy design (deservedly); what they'll miss is that its correctness is untested and one refactor away from regression. (Note: ARCC was not queried — MCP server unavailable; standard practices applied for the security-adjacent points above.)

### Confidence
`high` — the claims are grounded in direct reads of the actual files: absence of tests/CI/lint is verified by search, and every cited invariant, duplication, and type mismatch is in the quoted source.

# Synthesis — angles, not consensus

*(All five members are the same underlying model. Convergence below is a shared prior to stress-test, not independent confirmation.)*

## Shared starting points
Every member independently landed on the same four structural facts, each grounded in specific lines:

1. **The Postgres + Neo4j dual store is the central architectural liability** — no transactions, ordering enforced by comments, guaranteed drift ("cross-store ghosts" as the predicted #1 debugging dead-end).
2. **The live layer is single-process** (`liveEvents.ts` in-memory Map + in-memory rate-limit store) — a second Railway instance silently breaks the live wall.
3. **The per-IP rate limiter inverts at the target venue** — conference NAT means the whole event shares one bucket, while `X-Forwarded-For` spoofing lets a real abuser escape it.
4. **Zero tests/CI** — every carefully-commented invariant is one refactor from silent regression.

What might they *all* be missing for the same reason? They share a bias toward "collapse the stack now." None seriously costed the counterfactual: if the compounding cross-event graph *is* the moat, a graph-native store might earn its keep once features like "who should I meet next" (multi-hop recommendations) arrive — and a later Postgres→Neo4j migration under load is harder than the panel implies. None evaluated app-store/compliance surface (Apple sign-in rules, privacy nutrition labels), operating cost, or offline/flaky-venue UX for the QR scan flow itself.

## Genuine tensions
- **Collapse vs. harden.** Simplicity and Devil's Advocate say drop Neo4j into a Postgres `connections` table before launch (one transaction replaces every ordering comment). Scalability implicitly disagrees on priority: its highest-leverage fixes (delta broadcasts, TTL cache, Redis fanout, user-keyed rate limits) work *within* the current architecture. Your situation — a launch event on a deadline — decides this: the Postgres consolidation is a few days that deletes a failure class; the broadcast fixes are what keep the demo alive. If you can only do one before the event, Scalability's list wins; the panel's overall lean is that you have time for both because the codebase is small.
- **Simplicity vs. defense in depth.** Security wants RLS policies as a database backstop (single-layer app authz over a service-role key is the biggest structural risk); Simplicity's ethos would call RLS added complexity. Consolidating to Postgres actually resolves the tension — one store makes both the transaction story and RLS practical.
- **Frictionless as feature vs. vulnerability.** The product bet is one-scan connection; Security's sharpest point is that the scan is the *sole* authorization event and already over-grants (the group-route phone-number leak). Growth and privacy pull directly against each other here; the visibility model exists and is good — it just isn't uniformly enforced.
- **Novelty.** Devil's Advocate is the bear: "Linktree-QR + a party-trick wall" in a graveyard of dead predecessors, arguing the only defensible asset is the *compounding cross-event graph* — which the single-`eventId`-per-edge schema structurally cannot represent. No member disputes the graph-viz gap vs. Popl/Blinq/Luma is real; the disagreement is whether the current schema can cash the thesis.

## Blind spots (caught by exactly one member)
- **Security only:** group routes bypass the visibility filter and leak phone numbers on a single scan; `javascript:` URLs survive Zod's `.url()` into web-card hrefs; the feedback honeypot is dead code; anyone can inject edges into an event graph via caller-supplied `eventId`.
- **Devil only:** events with null `ends_at` are *permanent* unauthenticated public attendee lists; a rescan at event B steals the edge from event A's subgraph; `waft.app` vs `getwaft.app` config drift; Discord "groups" are channels visible to the whole shared hub guild.
- **Scalability only:** the web event page has no WS reconnect (and its cleanup closure is never wired in), so one WiFi blip permanently freezes the wall; `POST /events` blocks on a synchronous LLM call.
- **Maintainability only:** `packages/shared` types are camelCase fiction over snake_case reality and mobile hand-redeclares them — client/server drift is caught by runtime crashes on phones.
- **Nobody covered:** app-store review risk, GDPR beyond a mention, infra cost at event scale, analytics/retention instrumentation for the growth question, and what happens to the QR flow on zero-connectivity venue WiFi.

## Suggested direction
Sequenced for a launch-event deadline, supported by the named lenses:

1. **Fix the authorization leaks now** (Security): add the visibility filter to `groups.ts` queries, decide deliberately whether event graphs are public (and if so make null-`ends_at` walls expire), allow-list `http(s):` on rendered social URLs, and validate `eventId` attendance on `/connections`. These are hours, not days, and they're live PII exposure.
2. **Make the live path survive the demo** (Scalability): coalesced/delta broadcasts, a 2–5s TTL cache on `getEventGraph`, user-keyed rate limiting on authenticated routes, WS reconnect with jittered backoff on web, and stay explicitly single-instance until fanout moves to Redis/Supabase Realtime.
3. **Consolidate to Postgres** (Simplicity + Devil, with Maintainability's drift findings as evidence): one `connections` table + recursive CTE replaces every Cypher query in the repo, restores transactions, enables RLS as the backstop Security wants, and — critically for the growth thesis — lets you fix the schema so an edge can belong to *multiple* events (a join table), making the cross-event graph actually compound. This is the panel's strongest combined recommendation and also its most Claude-monoculture one — pressure-test it against a real future feature list before committing.
4. **Pin the invariants in tests** (Maintainability): ~10 vitest tests for visibility filtering, deletion ordering/idempotency, `card_code` preservation, and wall expiry. The specs already exist as comments.

**Residual uncertainty:** whether RLS/policies exist in the Supabase dashboard outside version control (all members could only see `infra/schema.sql`); whether Neo4j is load-bearing for a near-term roadmap feature the code doesn't show; and the novelty question is genuinely contested — the graph-wall demo is a real differentiator at events, but the bear case (schema can't compound the graph; group creation is deep-link theater) is grounded in code, not vibes.
